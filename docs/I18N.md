# Internationalization (i18n)

Last updated: 2026-05-18

## v1 Scope (SD-K-032)

**Spanish coverage is limited to customer-facing surfaces for v1.** Owner portal,
stylist portal, and admin/Command Center are English-only.

## Supported Locales

- `en-US` — English (default, fallback for everything)
- `es-MX` — Spanish (Mexico)
- `es-US` — Spanish (US Latino market)

## Locale Detection Priority

For every authenticated request:
1. `User.locale` (per-user preference)
2. `Organization.defaultLocale` (org-wide setting)
3. `Accept-Language` header (parsed against supported locales)
4. `en-US` (final fallback)

For unauthenticated requests (customer booking page), only steps 2-4 apply,
and step 2 is resolved via the organization that owns the booking page slug.

## Namespace Structure

Translations are organized by namespace:

| Namespace  | Purpose                                | Spanish coverage (v1) |
|------------|----------------------------------------|------------------------|
| `common`   | Shared strings (buttons, labels)       | YES                    |
| `customer` | Customer-facing booking/checkout pages | YES                    |
| `owner`    | Owner portal                           | English only           |
| `stylist`  | Stylist portal                         | English only           |
| `admin`    | Platform admin / Command Center        | English only           |

## File Layout

```
messages/
├── en-US/
│   ├── common.json
│   ├── customer.json
│   ├── owner.json
│   ├── stylist.json
│   └── admin.json
├── es-MX/
│   ├── common.json    (translated)
│   └── customer.json  (empty for v1 — fill per phase)
└── es-US/
    ├── common.json    (empty — falls back to es-MX, then en-US)
    └── customer.json  (empty — falls back to es-MX, then en-US)
```

## Adding a String

1. Add the key to `messages/en-US/{namespace}.json`
2. If customer-facing, also add to `messages/es-MX/customer.json` (or
   `messages/es-MX/common.json` for shared strings)
3. Use in code:

### Server component

```typescript
import { getTranslations } from 'next-intl/server';

export default async function Page() {
  const t = await getTranslations('common');
  return <h1>{t('welcome')}</h1>;
}
```

### Client component

```typescript
'use client';
import { T } from '@/lib/i18n/component';

export default function Button() {
  return <button><T id="common:save" /></button>;
}
```

### With interpolation

```typescript
<T id="customer:thanks" values={{ salon_name: 'Salon Envy' }} />
```

## Formatting Utilities

Always use these instead of manual formatting:

```typescript
import { formatCurrency, formatDate, formatRelativeTime } from '@/lib/i18n/format';

formatCurrency(2900, locale);          // "$29.00"
formatDate(new Date(), locale);        // "May 18, 2026"
formatRelativeTime(past, locale);      // "2 hours ago" / "hace 2 horas"
```

## Adding a New Locale

1. Add to `SUPPORTED_LOCALES` in `lib/i18n/config.ts`
2. Create `messages/{new-locale}/{namespace}.json` files
3. Update `parseAcceptLanguage` to handle the language code if needed
4. Update this doc

## Translation Workflow

For each customer-facing PR that adds new strings:
1. Developer adds English strings to `messages/en-US/customer.json`
2. Developer adds Spanish placeholders or initial translation to
   `messages/es-MX/customer.json` (can be the English string as a TODO)
3. Translation review happens in a follow-up PR before customer-facing
   feature ships
4. `es-US/customer.json` overrides only where Spanish (Mexico) variant
   differs noticeably from US Spanish

## Why next-intl?

- Native App Router support (Next 16 compatible)
- Server component compatible (`getTranslations` in RSC)
- ICU MessageFormat support for plurals/genders/etc.
- Tree-shakeable per namespace
- No URL-based locale routing needed (we use server-side detection)

**next-i18next (the legacy choice) does NOT support App Router. Don't use it.**
